## Pattern Overview

**Overall:** Next.js App Router monolith deployed on Vercel Hobby, with Supabase as primary database/storage backend.

**Key Characteristics:**
- Server-side rendered news portal with ISR caching (1-hour revalidate)
- Multi-theme system driven by a DB setting (`cp-site-type`) that switches between `netpro`, `insightkorea`, and `culturepeople` themes at runtime
- Admin CMS panel (`/cam/*`) with role-based access (superadmin, admin, reporter)
- API route handlers serve as the backend layer (no separate backend service)
- Client DB access layer (`src/lib/db.ts`) wraps all data operations through `/api/db/*` routes
- Server DB access layer (`src/lib/db-server.ts`) supports a 3-tier fallback: Supabase -> MySQL -> JSON file DB

## Layers

**Presentation Layer (Public):**
- Purpose: Renders the public-facing news portal
- Location: `src/app/page.tsx`, `src/app/article/[id]/`, `src/app/category/[slug]/`, `src/app/search/`, `src/app/tag/[name]/`
- Contains: Server components that fetch articles via `db-server.ts`, theme-switched layouts
- Depends on: `src/lib/db-server.ts`, `src/components/themes/`, `src/components/ui/`
- Used by: End users (readers)

**Presentation Layer (Admin CMS):**
- Purpose: Full CMS for article management, settings, analytics, and automation
- Location: `src/app/cam/`
- Contains: Client components with sidebar navigation, 25+ management pages
- Depends on: `src/lib/db.ts` (client DB layer), `src/lib/auth.ts`
- Used by: Admins and reporters

**API Layer:**
- Purpose: RESTful endpoints for all data operations
- Location: `src/app/api/`
- Contains: Route handlers for DB CRUD, authentication, cron jobs, AI, uploads, mail, RSS, SEO
- Depends on: `src/lib/db-server.ts`, `src/lib/cookie-auth.ts`, `src/lib/supabase-server-db.ts`
- Used by: Client components (via fetch), cron jobs, external integrations

**Server DB Layer:**
- Purpose: Unified data access with multi-backend fallback
- Location: `src/lib/db-server.ts`
- Contains: Functions prefixed `server*` (e.g., `serverGetArticles`, `serverCreateArticle`)
- Depends on: `src/lib/supabase-server-db.ts`, `src/lib/mysql-db.ts`, `src/lib/file-db.ts`
- Used by: API route handlers, server components

**Client DB Layer:**
- Purpose: Client-side data access through API proxy
- Location: `src/lib/db.ts`
- Contains: Functions like `getArticles`, `createArticle`, `getSetting`, `saveSetting`
- Depends on: `/api/db/*` routes via `fetch`
- Used by: Admin CMS pages (`src/app/cam/`)

**Authentication Layer:**
- Purpose: Cookie-based auth with HMAC-signed tokens
- Location: `src/lib/cookie-auth.ts`, `src/lib/auth.ts`, `src/middleware.ts`
- Contains: Token generation/verification, timing-safe comparison, token blacklist
- Depends on: `COOKIE_SECRET` env var, Web Crypto API
- Used by: Middleware, API routes, admin layout

**Theme System:**
- Purpose: Switchable UI themes for the public portal
- Location: `src/components/themes/culturepeople/`, `src/components/themes/insightkorea/`
- Contains: Landing, Article, Category, Header, Footer components per theme
- Depends on: `src/lib/site-type.ts` to determine active theme
- Used by: `src/app/page.tsx`, `src/app/article/[id]/page.tsx`, `src/app/category/[slug]/page.tsx`

**API Common (Registry):**
- Purpose: Shared services for the component registry (original landing page template system)
- Location: `src/app/api/_common/`
- Contains: `services/` (registry, search, page-registry, code-reader), `utils/` (errors, constants), `types/`
- Used by: `/api/v1/*` routes

## Data Flow

**Article Read (Public):**

1. User visits `/article/123` (numeric = article `no`, UUID = article `id`)
2. Server component calls `serverGetArticleByNo(123)` via `src/lib/db-server.ts`
3. `db-server.ts` checks `isSupabaseEnabled()` -> delegates to `src/lib/supabase-server-db.ts`
4. Supabase REST API fetches article with `revalidate: 60` cache tag
5. Theme is resolved via `getSiteType()` -> appropriate theme component renders the article
6. ISR caches the page for 3600 seconds

**Article Write (Admin):**

1. Admin edits article in `/cam/articles/[id]/edit`
2. Client component calls `updateArticle(id, updates)` from `src/lib/db.ts`
3. `db.ts` sends PATCH to `/api/db/articles` with credentials cookie
4. Middleware (`src/middleware.ts`) verifies `cp-admin-auth` cookie via `getAuthState()`
5. API route handler calls `serverUpdateArticle()` -> Supabase REST API with service key
6. `revalidateTag("articles")` invalidates ISR cache

**Cron Auto-News:**

1. Vercel Cron triggers `GET /api/cron/auto-news` daily at 21:00 UTC
2. Middleware validates `CRON_SECRET` Bearer token with rate limiting
3. Route handler fetches RSS sources, filters by keywords, calls AI for rewriting
4. Creates articles via `serverCreateArticle()` with deduplication check

**State Management:**
- No client-side state management library (no Redux, Zustand, etc.)
- Server state via ISR + `unstable_cache` with tag-based revalidation
- Client state via React `useState`/`useEffect` in admin pages
- Settings stored in DB as key-value pairs (`cp-*` prefixed keys)
- Auth state: HttpOnly cookie (`cp-admin-auth`) + localStorage for display name

## Key Abstractions

**Article:**
- Purpose: Core content entity with full lifecycle (draft -> review -> publish -> soft delete)
- Definition: `src/types/article.ts` - `Article` interface with 20+ fields
- Statuses: `"게시"` | `"임시저장"` | `"예약"` | `"상신"` | `"승인"` | `"반려"`
- Pattern: Soft delete via `deletedAt` field, audit trail for review workflow

**Site Settings (Key-Value Store):**
- Purpose: Dynamic configuration stored in DB, accessed via `serverGetSetting<T>(key, fallback)`
- Keys: `cp-site-type`, `cp-categories`, `cp-seo-settings`, `cp-sns-settings`, `cp-ads-global`, `cp-site-settings`, etc.
- Pattern: Generic typed getter/setter with JSON serialization
- Used everywhere: Layout, pages, API routes, theme selection

**Theme Components:**
- Purpose: Complete UI component sets per theme
- Pattern: Each theme exports Landing, ArticlePage, CategoryPage, Header, Footer via barrel file (`index.ts`)
- Examples: `src/components/themes/culturepeople/CulturePeopleLanding.tsx`, `src/components/themes/insightkorea/InsightKoreaLanding.tsx`

**Registry Components:**
- Purpose: Original landing page section components (component showcase system)
- Pattern: Each component in `src/components/registry/[name]/` with its own directory
- Examples: `src/components/registry/culturepeople-header-0/`, `src/components/registry/culturepeople-footer-6/`

## Entry Points

**Root Layout:**
- Location: `src/app/layout.tsx`
- Responsibilities: Font loading, Google Analytics/AdSense/Naver Analytics/Kakao SDK injection, ISR revalidate=3600, Providers wrapper, FloatingAds for non-admin pages

**Home Page:**
- Location: `src/app/page.tsx`
- Triggers: `GET /`
- Responsibilities: Fetches articles + site type + categories, renders theme-appropriate landing

**Article Page:**
- Location: `src/app/article/[id]/page.tsx`
- Triggers: `GET /article/:id` (numeric `no` or UUID `id`)
- Responsibilities: Article rendering with theme switching, SEO metadata, sidebar, comments, ads

**Admin Panel:**
- Location: `src/app/cam/layout.tsx` (client component)
- Triggers: `GET /cam/*`
- Responsibilities: Auth check, sidebar navigation, role-based menu filtering, activity logging

**Middleware:**
- Location: `src/middleware.ts`
- Triggers: All matched routes (API + admin paths)
- Responsibilities: Bot blocking, auth enforcement, RBAC, rate limiting, CRON_SECRET validation

## Error Handling

**Strategy:** Defensive try-catch at every boundary with graceful degradation

**Patterns:**
- API routes: `try { ... } catch { return NextResponse.json({ success: false, error: message }, { status: 5xx }) }`
- Server components: `try { ... } catch { console.error("[Context] message:", e); }` with fallback to empty data
- Client DB layer (`src/lib/db.ts`): 401 detection -> auto-redirect to `/cam/login?expired=1` with dedup flag
- Fire-and-forget for non-critical operations (view counts, activity logs) - errors silently ignored
- `fetchWithRetry()` (`src/lib/fetch-retry.ts`): Exponential backoff for 5xx errors, max 2 retries

## Caching Strategy

**ISR (Incremental Static Regeneration):**
- Root layout: `revalidate = 3600` (1 hour)
- Home page: `revalidate = 3600`
- Article page: `revalidate = 3600`
- Tag-based revalidation: `revalidateTag("articles")` on write operations

**Supabase Query Caching:**
- Article list queries: `next: { revalidate: 60, tags: ["articles"] }`
- Single article queries: `next: { revalidate: 60, tags: ["articles"] }`

**Vercel Edge Caching (vercel.json):**
- Static assets (`/_next/static/*`): `max-age=31536000, immutable`
- Fonts: `max-age=31536000, immutable`
- RSS feed: `s-maxage=300, stale-while-revalidate=600`
- OG images: `s-maxage=86400, stale-while-revalidate=604800`
- Sitemap/robots.txt: `no-store, no-cache, must-revalidate`

**Admin Pages:**
- `/cam/*` headers: `no-store, max-age=0` (never cached)

## Cross-Cutting Concerns

**Authentication:**
- HMAC-signed tokens in HttpOnly cookie (`cp-admin-auth`), 24-hour expiry
- Middleware enforces auth on all `/api/db/*`, `/api/admin/*`, `/api/ai/*`, `/cam/*` paths
- CRON_SECRET Bearer token for server-to-server and cron calls with per-IP rate limiting (5/min)
- API v1 uses HTTP Basic Auth for external consumers

**Authorization (RBAC):**
- Three roles: `superadmin`, `admin`, `reporter`
- Reporter restricted to `/cam/login`, `/cam/dashboard`, `/cam/articles` paths
- Enforced in middleware (`src/middleware.ts` lines 176-183) and admin layout menu filtering

**Security Headers (CSP, HSTS, etc.):**
- Configured in `next.config.ts` `headers()` function
- CSP allows inline scripts (for analytics), AdSense, Google Tag Manager, Coupang, Naver
- HSTS enabled with `includeSubDomains`
- Admin pages: X-Frame-Options DENY (anti-clickjacking)
- Permissions-Policy blocks camera/microphone/geolocation

**Bot Blocking:**
- Middleware blocks 16 AI/scraper bots via User-Agent regex
- Allows search engine bots and AI search answer bots (ChatGPT-User, PerplexityBot)

**Logging:**
- `console.error`/`console.warn` for server-side errors with `[context]` prefixes
- Activity log: `src/lib/log-activity.ts` -> `POST /api/db/activity-logs` (fire-and-forget from client)
- Access logs: `src/app/api/db/access-logs/`
- View logs: `src/app/api/db/view-logs/`

**Validation:**
- Zod for site config validation (`src/config/site.tsx`)
- Manual validation in API routes (search query length limits, numeric ID bounds)
- DOMPurify for HTML sanitization in article bodies

**Input Sanitization:**
- Search queries truncated to 200 characters
- Article IDs: numeric limited to 9 digits to prevent parseInt overflow
- Timing-safe string comparison for all secret comparisons (`timingSafeEqual`)
