## Critical Concerns

### In-Memory State in Serverless Environment
- Issue: Multiple modules use in-memory `Map`/`Set` for rate limiting and token blacklists, which do not persist across serverless function invocations on Vercel.
- Files:
  - `src/lib/cookie-auth.ts` (lines 2-16): `tokenBlacklist` Set with `setInterval` cleanup. The `setInterval` never fires reliably in serverless; the blacklist resets on each cold start.
  - `src/middleware.ts` (lines 8-25): `cronRateLimitMap` in-memory Map. Each middleware invocation may get a fresh instance.
  - `src/app/api/auth/login/route.ts` (lines 42-43): `memAttempts` Map for login rate limiting fallback. Ineffective if Redis is not configured.
  - `src/app/api/db/comments/route.ts` (lines 64-93): `commentRateMap` in-memory Map.
- Impact: Token invalidation on logout does not work reliably. Rate limiting is bypassed across cold starts. Login brute-force protection fails without Redis.
- Fix approach: Use Upstash Redis (already a dependency) for all rate limiting and token blacklists. Remove in-memory fallbacks or clearly mark them as dev-only.

### 19 Known Dependency Vulnerabilities (14 High)
- Issue: `pnpm audit` reports 19 vulnerabilities including 14 high severity.
- Key packages:
  - `axios@1.13.2` - DoS via `__proto__` key in mergeConfig (patch: >=1.13.5)
  - `shadcn@3.8.5` transitive: `@isaacs/brace-expansion`, `minimatch`, `diff` - ReDoS/resource consumption
- Impact: `axios` is used in production code (`src/app/api/` routes). The shadcn vulnerabilities are devDependencies only.
- Fix approach: Update `axios` to >=1.13.5 immediately. Update `shadcn` when a patched version is available.

## Security

### ESLint Rules Disabled for Type Safety
- Issue: Critical TypeScript ESLint rules are globally disabled in `eslint.config.mjs`:
  - `@typescript-eslint/no-explicit-any`: off
  - `@typescript-eslint/no-unused-vars`: off
  - `@typescript-eslint/no-empty-object-type`: off
- Files: `eslint.config.mjs` (lines 16-19)
- Impact: `any` types suppress compile-time checks. Unused variables indicate dead code. Both increase risk of runtime errors.
- Fix approach: Re-enable rules at `warn` level. Fix violations incrementally. Keep `off` only for `src/components/registry/` which has its own relaxed config.

### Cookie Secure Flag Conditional on NODE_ENV
- Issue: Auth cookie `secure` flag is set only when `NODE_ENV === "production"`. If `NODE_ENV` is misconfigured, cookies transmit over HTTP.
- Files: `src/app/api/auth/login/route.ts` (lines 203-206, 251-253)
- Impact: Low risk since Vercel enforces HTTPS, but defense-in-depth is weakened.
- Fix approach: Always set `secure: true` in production deployments regardless of `NODE_ENV`.

### SMTP Credentials Stored in Settings DB
- Issue: SMTP credentials (host, user, password) are stored in the `cp-smtp-settings` settings key in Supabase. While password is encrypted via `src/lib/encrypt.ts`, the encryption key derives from `COOKIE_SECRET`.
- Files: `src/app/cam/settings/page.tsx` (lines 14-31), `src/lib/encrypt.ts`
- Impact: If `COOKIE_SECRET` is compromised, SMTP credentials are exposed. Settings table has RLS but service key access exposes all values.
- Fix approach: Store SMTP credentials as Vercel environment variables instead of DB settings. Use a separate encryption key for sensitive DB values.

### `unsafe-inline` and `unsafe-eval` in CSP
- Issue: Content Security Policy includes `'unsafe-inline'` and `'unsafe-eval'` for scripts due to AdSense/analytics requirements.
- Files: `next.config.ts` (line 26)
- Impact: Reduces XSS protection from CSP. Necessary trade-off for third-party ad scripts.
- Fix approach: Consider using nonce-based CSP for first-party scripts while keeping `unsafe-inline` only for third-party domains. Requires Next.js CSP nonce support.

## Performance

### Full Table Scans via `serverGetArticles()`
- Issue: `serverGetArticles()` fetches ALL articles (paginated 1000 at a time from Supabase) without body content. Called 20+ times across the codebase including sitemap generation, RSS feeds, page rendering, cron jobs, and admin APIs.
- Files:
  - `src/lib/db-server.ts` (line 21): Core function
  - `src/lib/supabase-server-db.ts` (lines 80-117): Implementation fetches all rows in a loop
  - Called from: `src/app/page.tsx`, `src/app/sitemap.xml/route.ts`, `src/app/api/rss/route.ts`, `src/app/api/cron/auto-press/route.ts`, `src/app/api/cron/auto-news/route.ts`, and 15+ more files
- Impact: As article count grows (currently ~4000), each call transfers increasing amounts of data. Multiple calls on the same page compound the issue. Supabase REST API pagination adds latency per 1000-row page.
- Fix approach: Add purpose-specific queries (e.g., `serverGetRecentArticles(limit)`, `serverGetArticleIds()`). Use `unstable_cache` with appropriate tags to avoid redundant fetches. Add DB-level filtering (status, date range) instead of fetching all then filtering in JS.

### Client-Side Filtering of All Articles
- Issue: The GET handler for `/api/db/articles` fetches all articles, then filters in memory for search (`q`), category, and status.
- Files: `src/app/api/db/articles/route.ts` (lines 50-88)
- Impact: For search queries, all articles are loaded from DB even when only a few match. No DB-level text search is used for the main articles API.
- Fix approach: Push filters to the Supabase query level. Use `ilike` for text search or the existing `sbSearchArticles` function.

### No Image Optimization
- Issue: `images.unoptimized: true` in Next.js config disables all image optimization due to Vercel Hobby plan limits.
- Files: `next.config.ts` (line 42)
- Impact: All images served at original size/format. No WebP conversion, no responsive sizing. Affects page load performance for readers.
- Fix approach: Use external image CDN (e.g., Cloudflare Images, imgproxy) or upgrade Vercel plan. Alternatively, optimize images at upload time in `src/app/api/upload/image/route.ts`.

## Maintainability

### Triple DB Fallback Pattern Everywhere
- Issue: Every data access function in `src/lib/db-server.ts` follows the same pattern: try Supabase -> catch and try MySQL -> catch and try file-db. This is repeated 15+ times.
- Files: `src/lib/db-server.ts` (458 lines), `src/lib/supabase-server-db.ts` (424 lines), `src/lib/mysql-db.ts` (237 lines), `src/lib/file-db.ts` (109 lines) - total 1228 lines for data access.
- Impact: Hard to maintain. Bugs must be fixed in three places. MySQL and file-db fallbacks are unused in production (Supabase is the only backend).
- Fix approach: Remove MySQL and file-db fallbacks if Supabase is the sole production backend. Keep them only if actively used for local development. Simplify `db-server.ts` to a single Supabase implementation.

### No Unit Tests
- Issue: Zero unit test files exist in `src/`. The only tests are E2E API tests in `e2e/` (14 test files) that test the registry API, not the news portal features.
- Files: `e2e/api/*.test.ts` - all test registry/component endpoints, none test articles, auth, comments, or cron jobs.
- Impact: No automated verification of core business logic (article creation, auth flow, image migration, search). Regressions are caught only in production.
- Fix approach: Add unit tests for critical paths: `src/lib/cookie-auth.ts` (token generation/verification), `src/lib/db-server.ts` (data access), `src/app/api/db/articles/route.ts` (CRUD operations), `src/app/api/auth/login/route.ts` (authentication flow).

### Large Monolithic Admin Pages
- Issue: Admin pages are single-file components exceeding 700-1100 lines with mixed concerns (state management, API calls, rendering, validation).
- Files:
  - `src/app/cam/settings/page.tsx`: 1167 lines
  - `src/app/cam/articles/[id]/edit/page.tsx`: 934 lines
  - `src/app/cam/ai-settings/page.tsx`: 876 lines
  - `src/app/cam/auto-press/page.tsx`: 848 lines
  - `src/app/cam/articles/upload-md/page.tsx`: 845 lines
- Impact: Difficult to understand, modify, or debug. State management is ad-hoc with many `useState` calls.
- Fix approach: Extract reusable hooks (e.g., `useArticleForm`, `useSettings`). Split into smaller sub-components. Consider using a form library consistently.

### Duplicate Supabase Client Logic in Comments Route
- Issue: `src/app/api/db/comments/route.ts` creates its own Supabase REST client (`sbHeaders`, direct fetch calls) instead of using `src/lib/supabase-server-db.ts`.
- Files: `src/app/api/db/comments/route.ts` (lines 8-35)
- Impact: Two separate implementations for Supabase access. Bug fixes in one location don't propagate.
- Fix approach: Move comment DB operations to `src/lib/supabase-server-db.ts` and call from the route handler.

3. **[High]** Add unit tests for `cookie-auth.ts`, `db-server.ts`, and article API routes. Start with auth token verification and article CRUD.
4. **[High]** Replace `serverGetArticles()` full-table scans with purpose-specific queries. Add DB-level filtering for the articles API.
5. **[Medium]** Clean up root directory: delete 34+ temp files, add `.gitignore` patterns.
6. **[Medium]** Remove or archive one-off migration scripts and admin API endpoints.
7. **[Medium]** Consolidate Supabase client code: remove duplicate in comments route, eliminate MySQL/file-db fallbacks if unused in production.
8. **[Medium]** Re-enable `@typescript-eslint/no-explicit-any` at warn level and fix violations incrementally.
9. **[Low]** Refactor large admin pages (1000+ lines) into smaller components with extracted hooks.
10. **[Low]** Evaluate separating registry components (14MB, 1014 directories) into a separate repository.
